# 想法拆解器 (Idea Unblocker)

把纸上的想法变成可执行的下一步。核心哲学：**不追求完美规划，只追求"今天能启动"。**

## 触发条件

以下任何一种情况都应触发此skill：

1. **图片触发**：用户发来手写便签、笔记本页面、todo list 的照片
2. **文字触发**：用户提到"拆解"、"怎么推进"、"卡住了"、"下一步"、"整理想法"等
3. **想法描述**：用户描述一个想法并希望知道怎么开始

## 处理流程

### Step 1：识别 & 提取

**如果是图片输入：**
- 识别照片中的所有条目
- 自动标记状态：
  - ✓ / ✔ / 勾 → 已完成，跳过
  - ✗ / 划线 / 涂掉 → 已放弃，跳过
  - 无标记 → 未完成，进入分诊
- 对于识别不确定的文字，用「XX（？）」标注并询问用户确认
- 区分中英文混合内容（用户经常混用）

**如果是文字输入：**
- 直接提取想法，跳到 Step 2

### Step 2：快速分诊（全局鸟瞰）

对所有未完成的条目，生成一张分诊表：

```
| # | 想法 | 领域 | 链条 | 卡在哪 | 建议下一步（15min内可做）|
|---|------|------|------|--------|------------------------|
```

**领域分类**（基于用户的实际工作领域）：
- 投资研究
- 技术/工具
- 内容创作
- 生物英雄传
- 个人/生活
- Network State

**链条长度判断标准：**
- ⚡ 短（≤1天）：1-3步可完成，不依赖他人
- 🔗 中（1天-1周）：需要多个session，可能有外部依赖
- 🏗️ 长（>1周）：需要持续投入，涉及多个子项目

**"卡在哪"诊断**（这是核心价值，常见卡点类型）：
- **范围不明确**：想法太大太模糊，不知从哪切入
- **输出形式未定**：知道想做什么但不知道做成什么样
- **技术路径不清**：知道目标但不知道用什么工具/方法
- **外部依赖**：需要等别人、等资源、等时机
- **启动阻力大**：其实知道怎么做但链条太长不想开始
- **优先级模糊**：不确定现在该不该做这个

**"建议下一步"撰写规则：**
- 必须包含明确的动作动词（搜索、写、列、问、测试、注册...）
- 必须可在15分钟内完成
- 必须不依赖其他任务先完成
- 对于"启动阻力大"的任务，刻意给一个极小的第一步

### Step 3：用户选择深拆

分诊表展示后，询问用户想深入拆解哪几个。不要自动全部拆解——大部分想法只需要一个清晰的下一步就够了。

**深拆时的规则：**
- 拆成 3-7 个具体步骤
- 每步 ≤ 60分钟（超过的继续拆）
- 按依赖关系排序
- 每步有明确动作动词
- 标注总预估时间
- 特别标出"今天的15分钟"——最小启动步骤

**深拆输出格式：**
```
## 🔨 [想法名称]

**链条长度：** 🔗 中 (1天-1周)
**总预估时间：** ~X小时
**卡点诊断：** [具体卡在哪]

**步骤：**
1. [动词开头的具体步骤]（Xmin）
2. [动词开头的具体步骤]（Xmin）
3. ...

**⚡ 今天的15分钟：** [最小启动步骤的具体描述]
```

### Step 4：写入 Notion（可选）

深拆完成后，询问用户是否要写入 Notion 想法收集箱。

**写入目标：** 💡想法收集箱 数据库
- data_source_id: `d3a7a9cb-dd0e-4761-85f3-f1692d91911b`
- 需要用 Notion MCP 工具 `notion-create-pages`

**字段映射：**
- 想法 → 想法名称
- 领域 → 从分诊表的领域列取值（投资研究/技术/工具/内容创作/生物英雄传/个人/生活/Network State）
- 链条长度 → ⚡ 短 (≤1天) / 🔗 中 (1天-1周) / 🏗️ 长 (>1周)
- 下一步动作 → 分诊表或深拆中的"建议下一步"
- 状态 → "Not started"
- 来源 → 根据情况选择（纸质笔记本/灵感闪现/阅读/播客/与人交流/Claude对话）
- 记录日期 → 当天日期

**写入时的 properties 格式：**
```json
{
  "想法": "想法名称",
  "领域": "技术/工具",
  "链条长度": "🔗 中 (1天-1周)",
  "下一步动作": "具体的下一步描述",
  "状态": "Not started",
  "来源": "纸质笔记本",
  "date:记录日期:start": "2026-03-27",
  "date:记录日期:is_datetime": 0
}
```

## 语气和风格

- 用中文回复（用户的工作语言）
- 简洁直接，不啰嗦
- 对"卡在哪"的诊断要诚实——如果一个想法其实不值得做，可以温和地指出
- 鼓励"做了再说"而不是"想清楚再做"
- 核心信念：**完成比完美重要，启动比规划重要**

## 注意事项

- 手写字识别不确定时要标注并确认，不要猜
- 已完成（✓）和已放弃（✗/划线）的条目要识别出来但不处理
- 不要一次性拆解所有想法——让用户选择聚焦哪几个
- 分诊表是快速扫描，不是深度分析——保持轻量
- 如果用户的想法明显相关可以合并，主动建议
